# 明远湖大豆包 — 使用与架构说明

## 系统概述

明远湖大豆包（DataFinderAgentOS）是一个基于 Tornado 异步 Web 框架的权限管理后台系统。核心功能包括用户管理、角色管理、功能权限管理和菜单管理，采用功能码与菜单导航分离的双体系 RBAC（基于角色的访问控制）模型。

- **后端框架**：Tornado（Python 异步 Web 框架）
- **数据库**：SQLite，WAL 模式
- **前端**：Layui 2.9.8
- **端口**：10010

---

## 快速开始

### 环境要求

- Python 3.11+
- 无需额外数据库安装（SQLite 内建）

### 安装与启动

```bash
pip install -r requirements.txt
python main.py
```

首次启动会自动完成：
1. 创建 `database/finderos.db` 数据库文件
2. 初始化七张数据表
3. 注入默认角色、管理员账号、功能权限码和菜单数据

启动后访问 `http://localhost:10010` 即可登录。

### 默认账号

| 用户名 | 密码 | 角色 |
|--------|------|------|
| admin | GOD-陌 | 系统管理员 |

管理员密码可通过环境变量 `ADMIN_PASS` 自定义。

## 功能模块

### 1. 用户认证

- **密码安全**：前端 SHA256 哈希 → 后端 PBKDF2-SHA256（10 万次迭代 + 随机盐值），数据库中仅存储密文
- **会话管理**：Tornado Secure Cookie，退出时清除
- **XSRF 防护**：所有表单和 AJAX 请求均带 XSRF Token
- **登录路由**：管理员 → `/admin` 后台首页，普通用户 → `/index` 前台首页

### 2. 用户管理（`/admin/user`）

- 用户列表，支持分页和用户名关键词搜索
- 新增用户：设置用户名、密码、所属角色
- 编辑用户：修改用户名、密码（留空不修改）、角色
- 启用/禁用用户（`admin` 内置账号禁止禁用）
- 删除用户（`admin` 内置账号禁止删除）

### 3. 角色管理（`/admin/role`）

- 角色列表，系统角色标记为不可编辑/不可删除
- 新增/编辑角色：名称、描述
- 功能权限分配：勾选该角色可执行的操作权限码
- 菜单分配：勾选该角色在左侧导航栏可见的菜单项
- 删除角色时自动解除关联用户的角色绑定

### 4. 功能权限管理（`/admin/function`）

- 功能权限码列表，支持分页
- 权限码格式：`模块:操作`，例如 `user:view`、`role:edit`
- 每个权限码包含：代码、名称、所属模块、图标、路由、父级、排序
- 支持层级结构（通过 parent_id）
- 启用/禁用权限码，禁用时自动清除角色关联
- 删除时级联清除角色-功能关联

### 5. 菜单管理（`/admin/menu`）

- 菜单列表，支持分页
- 菜单项包含：代码、名称、路径、图标、排序、父级
- 支持父子层级，构建左侧导航树
- 启用/禁用菜单，禁用时自动清除角色-菜单关联
- 删除菜单时级联删除子菜单和角色关联

### 6. 浏览器指纹采集

- 页面加载时自动采集浏览器特征并上报至 `/fp` 接口
- 采集维度：User-Agent、屏幕分辨率、色深、时区、语言、CPU 核心数、内存、触摸点、Canvas 指纹、WebGL 指纹、字体指纹、像素比等
- 生成唯一指纹哈希存入数据库 `fingerprints` 表
- 已登录用户自动关联用户 ID

### 7. 全局水印

- 所有页面均显示 "COD·陌" 半透明水印
- 覆盖全屏，不可选中，不可点击，不影响正常操作

### 请求处理流程

```
浏览器请求 → main.py 路由匹配 → Controller Handler → Repository → SQLite
                                         ↓
                                    模板渲染 (Tornado)
                                         ↓
                                    返回 HTML/JSON
```

### 数据库表结构

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| users | 用户表 | username, password_hash, salt, role_id, is_enabled |
| roles | 角色表 | name, description, is_system |
| functions | 功能权限码表 | code, name, module, parent_id, sort_order |
| menus | 菜单表 | code, name, path, icon, parent_id, sort_order |
| role_functions | 角色-功能关联表 | role_id, function_id（联合主键） |
| role_menus | 角色-菜单关联表 | role_id, menu_id（联合主键） |
| fingerprints | 浏览器指纹表 | fingerprint, user_agent, canvas_hash, webgl_hash 等 |

### 权限模型

```
用户 ──1:1──> 角色 ──M:N──> 功能权限码 (role_functions)
                 ──M:N──> 菜单       (role_menus)
```

两条独立的权限链路：

1. **功能权限码**（后端）：控制用户可以执行哪些操作，格式为 `模块:操作`，如 `user:view`、`role:edit`。管理员基类在每次请求时校验用户是否拥有对应权限码。

2. **菜单**（前端）：控制左侧导航栏显示哪些菜单项。菜单与功能权限码是两个独立体系，互不依赖。角色的可见菜单通过 `role_menus` 关联表控制。

### 管理员门控

所有 `/admin/*` 路由均经过五层检查（由 `AdminBaseHandler.prepare()` 实现）：

1. 是否登录
2. 账号是否存在且启用
3. 是否分配了角色
4. 角色是否为"普通用户"（普通用户禁止访问后台）
5. 角色是否拥有功能权限

## 安全机制

- 密码双哈希：客户端 SHA256 + 服务端 PBKDF2-SHA256（10 万轮，随机盐）
- 所有数据库操作使用参数化查询
- XSRF Token 保护所有表单提交
- Secure Cookie 管理会话
- 系统角色（`is_system=1`）禁止编辑和删除
- 内置 `admin` 用户禁止禁用和删除
